# GPT Action Setup Guide

This guide explains how to publish the API and connect it to a Custom GPT as a GPT Action.

---

## Overview

The API is designed to be consumed by a Custom GPT via OpenAI's GPT Actions feature. The Custom GPT calls these endpoints to gather real research data before generating ebook content. No content is generated by the API itself — all endpoints return structured planning data only.

---

## Step 1 — Publish the API

### 1.1 Set required secrets

Before deploying, confirm the following are set in **Replit Secrets** (the lock icon in the sidebar):

| Secret | Description |
|--------|-------------|
| `TAVILY_API_KEY` | Your Tavily API key from [app.tavily.com](https://app.tavily.com) |
| `SESSION_SECRET` | Any secure random string (reserved — not currently used by routes) |

`PORT` is injected automatically by Replit at deploy time. Do not add it as a secret.

### 1.2 Deploy the project

1. Open the project in Replit
2. Click the **Deploy** button in the top navigation bar
3. Choose a deployment type:
   - **Autoscale** — recommended for GPT Action use (scales to zero when idle, no fixed cost)
   - **Reserved VM** — use if you need guaranteed uptime with no cold starts
4. Click **Deploy** to start the build

Replit will:
- Run `pnpm --filter @workspace/api-server run build` to compile the server
- Start the server with `node --enable-source-maps artifacts/api-server/dist/index.mjs`
- Run a startup health check at `GET /api/healthz` — deployment only succeeds if this returns `200`

### 1.3 Confirm deployment

Once deployed, the health check endpoint should respond:

```
GET https://<your-app-name>.replit.app/api/healthz
→ { "status": "ok" }
```

## Step 2 — Obtain the Production URL

Your production base URL takes the form:

```
https://<your-app-name>.replit.app/api
```

To find the exact URL:
- Open the Replit deployment panel after deploying
- The full domain is shown under "Deployment URL"
- It is also available at runtime in the `REPLIT_DOMAINS` environment variable

All API endpoints are available at this base URL. Example:

```
https://<your-app-name>.replit.app/api/research
https://<your-app-name>.replit.app/api/outline
https://<your-app-name>.replit.app/api/chapter-research
```

---

## Step 3 — Update the OpenAPI Server URL

The OpenAPI spec currently uses a relative server URL (`/api`) which is correct for development but **GPT Actions require an absolute HTTPS URL**.

### 3.1 Edit the spec

Open `lib/api-spec/openapi.yaml` and update the `servers` block (lines 7–9):

**Before (development default):**
```yaml
servers:
  - url: /api
    description: Base API path
```

**After (production):**
```yaml
servers:
  - url: https://<your-app-name>.replit.app/api
    description: Production API
```

Replace `<your-app-name>` with your actual Replit deployment subdomain.

### 3.2 Important constraints

- **Do not change `info.title`** — it is currently `Api` and must stay that way. Changing it breaks the generated import paths in `lib/api-zod` and `lib/api-client-react`.
- **Do not edit generated files** in `lib/api-zod/src/generated/` or `lib/api-client-react/src/generated/` — they are overwritten by codegen.
- After changing the spec, re-run codegen if you plan to use the generated Zod schemas:
  ```bash
  pnpm --filter @workspace/api-spec run codegen
  ```
  This is only needed if new schema changes were made alongside the server URL update.

## Step 4 — Import the Schema into a Custom GPT

### 4.1 Open your Custom GPT

1. Go to [chat.openai.com](https://chat.openai.com)
2. Click your profile → **My GPTs** → **Create a GPT** (or edit an existing one)
3. Go to the **Configure** tab

### 4.2 Add a GPT Action

1. Scroll to **Actions** and click **Create new action**
2. In the **Schema** field, click **Import from URL** — or paste the YAML directly

**To import from URL** (recommended — always stays in sync):
```
https://<your-app-name>.replit.app/api/openapi.yaml
```
> Note: The API does not currently serve the spec file at a URL. Use the paste method below instead, or add a static file route to serve it.

**To paste directly:**
1. Open `lib/api-spec/openapi.yaml` in the Replit editor
2. Copy the entire file contents
3. Paste into the GPT Actions schema field
4. OpenAI will parse and validate it automatically

### 4.3 Verify the schema import

After importing, the GPT Actions editor should show all 8 endpoints listed:

| Method | Path | Name |
|--------|------|------|
| GET | `/healthz` | Health check |
| POST | `/research` | Research a topic for book creation |
| POST | `/outline` | Generate a research-based book outline |
| POST | `/citations` | Generate formatted academic citations |
| POST | `/kdp-research` | Research KDP publishing metadata |
| POST | `/kdp-keywords` | Research Amazon KDP keyword opportunities |
| POST | `/book-market-research` | Analyse a book topic's market viability |
| POST | `/chapter-research` | Deep research for a specific book chapter |

If any endpoint is missing, check that the server URL in the spec is the absolute production URL and that the YAML is valid.

## Step 5 — Authentication

### Current state

The API has **no authentication layer**. All endpoints are publicly accessible to any caller with the URL. CORS is enabled for all origins.

This means:
- No API key is required in the GPT Actions configuration
- Set **Authentication** to **None** in the GPT Actions editor
- Anyone who discovers the URL can call the API

### Recommended for production

If you want to restrict access to your Custom GPT only:

1. Add an API key middleware to the Express server (a single middleware in `app.ts` checking an `Authorization: Bearer <token>` header)
2. Add the key as a Replit Secret
3. In the GPT Actions editor, set **Authentication** to **API Key** and enter the same key

This is not currently implemented and is outside the current feature scope.

## Step 6 — Testing the GPT Action

### 6.1 Test via the GPT Actions editor

In the GPT Actions editor, click **Test** next to any endpoint. A successful test shows the raw JSON response from your production server.

**Test the health check first:**
- Method: `GET`
- Path: `/healthz`
- Expected: `{ "status": "ok" }`

**Test a research endpoint:**
- Method: `POST`
- Path: `/research`
- Body:
  ```json
  {
    "topic": "productivity",
    "book_type": "ebook",
    "target_audience": "remote workers"
  }
  ```
- Expected: JSON with `topic`, `summary`, `key_concepts`, `chapter_ideas`, `statistics`, `sources`, and `academic_references`

### 6.2 Test via curl

```bash
# Health check
curl https://<your-app-name>.replit.app/api/healthz

# Research
curl -X POST https://<your-app-name>.replit.app/api/research \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity","book_type":"ebook","target_audience":"remote workers"}'

# Outline
curl -X POST https://<your-app-name>.replit.app/api/outline \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity","book_type":"ebook","target_audience":"remote workers","length":"10000-20000"}'

# Chapter research
curl -X POST https://<your-app-name>.replit.app/api/chapter-research \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity","chapter_title":"Building a Morning Routine","target_audience":"remote workers"}'

# KDP keywords
curl -X POST https://<your-app-name>.replit.app/api/kdp-keywords \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity"}'

# KDP research
curl -X POST https://<your-app-name>.replit.app/api/kdp-research \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity","book_type":"ebook","target_audience":"remote workers"}'

# Book market research
curl -X POST https://<your-app-name>.replit.app/api/book-market-research \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity"}'

# Citations
curl -X POST https://<your-app-name>.replit.app/api/citations \
  -H "Content-Type: application/json" \
  -d '{"topic":"productivity","style":"APA"}'
```

### 6.3 Common issues and fixes

| Symptom | Cause | Fix |
|---------|-------|-----|
| GPT Action shows `fetch error` or no response | Server URL in spec is still `/api` (relative) | Update `servers.url` to the absolute production URL |
| All Tavily endpoints return 500 | `TAVILY_API_KEY` is missing or incorrect | Check Replit Secrets; restart the deployment after adding the secret |
| `400 Bad Request` | Request body is missing a required field | Check the request schema — all fields marked required must be present |
| GPT sends wrong field names | Schema import didn't update after spec change | Re-import the schema from the updated YAML in the GPT Actions editor |
| Cold start delay on first request | Autoscale deployment scaling up from zero | Switch to Reserved VM, or prime the server with a health check before calling data endpoints |
| `{ "status": "ok" }` on healthz but 500 on POST | `TAVILY_API_KEY` not present at runtime | The secret must be set before deploy, not after. Redeploy after adding the secret. |
