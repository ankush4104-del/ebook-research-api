# PROJECT_STATUS.md

## 1. Project Overview

A research-focused REST API server designed as the backend for a Custom GPT that generates publishing-ready ebooks. All endpoints return structured planning and research data only — **no content is generated by the API**. The Custom GPT calls these endpoints as GPT Actions to gather real data before writing.

---

## 2. Architecture Summary

```
pnpm monorepo — Node.js 24, TypeScript 5.9, Express 5
├── artifacts/api-server/     — The live Express API (the only deployable service)
├── lib/api-spec/             — ⭐ OpenAPI 3.1 spec (single source of truth for all schemas)
├── lib/api-zod/              — Zod validators generated from the spec (consumed by api-server)
├── lib/api-client-react/     — React Query hooks generated from the spec (unused — for future UI)
└── lib/db/                   — Drizzle ORM + PostgreSQL schema (present but not yet used)
```

**Request lifecycle:**
1. Client sends a POST request to `/api/<endpoint>`
2. Express route validates the request body using a Zod schema from `@workspace/api-zod`
3. Route calls the corresponding lib function in `src/lib/`
4. Lib function queries Tavily (web research) and/or Crossref (academic research)
5. Structured JSON response is returned

**Contract-first workflow:**
- All schemas are defined in `lib/api-spec/openapi.yaml` first
- `pnpm --filter @workspace/api-spec run codegen` regenerates Zod schemas and React Query hooks
- Routes import validated Zod bodies from `@workspace/api-zod`
- Generated files in `lib/api-zod/src/generated/` and `lib/api-client-react/src/generated/` must never be edited manually

## 3. Tavily Integration

- **API endpoint:** `https://api.tavily.com/search`
- **Auth:** `api_key` field in JSON body (from `TAVILY_API_KEY` environment variable)
- **Used by:** `/research`, `/outline`, `/kdp-research`, `/kdp-keywords`, `/book-market-research`, `/chapter-research`
- **Search depth:** `advanced` for most endpoints; `basic` for lighter lookups within `/research`
- **Max results:** 7–10 per search; most endpoints run 2–4 parallel Tavily searches
- **Error handling:** Throws on API failure with a descriptive error message; unhandled exceptions return HTTP 500

---

## 4. Crossref Integration

- **API endpoint:** `https://api.crossref.org/works`
- **Auth:** None required (public API)
- **Required header:** `User-Agent: BookResearchAPI/1.0 (mailto:research@example.com)` — required for Crossref polite pool access
- **Used by:** `/citations`, `/chapter-research`, `/research`
- **Query params:** `query`, `rows` (5–10), `select` (field projection)
- **Filter:** Only items with both a `title` and a `DOI` are used
- **Error handling:** Returns empty array on failure; does not block the response

---

## 5. OpenAPI Integration

- **Spec location:** `lib/api-spec/openapi.yaml`
- **Version:** OpenAPI 3.1.0
- **Codegen tool:** Orval (`lib/api-spec/orval.config.ts`)
- **Codegen command:** `pnpm --filter @workspace/api-spec run codegen`
- **Outputs:**
  - `lib/api-zod/src/generated/api.ts` — Zod request/response schemas (consumed by the server)
  - `lib/api-client-react/src/generated/` — React Query hooks (unused; for future UI)
- **GPT Actions:** The spec is compatible with OpenAI GPT Actions. The `servers.url` must be updated to the absolute production URL before importing into a Custom GPT.
- **`info.title`:** Must remain `Api` — changing it breaks generated import paths
